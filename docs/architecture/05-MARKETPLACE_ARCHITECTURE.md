# PeakVox — Marketplace Architecture

**Owns:** the Voice Marketplace — publishing, discovery, preview, usage, and the
royalty-on-use loop that ties the marketplace to monetization. **Cloud-only; schema-ready in
CE.**

> See also: [Domain](02-DOMAIN_ARCHITECTURE.md) · [Monetization](07-MONETIZATION_ARCHITECTURE.md)
> · [Data §4](03-DATA_ARCHITECTURE.md)

---

## 1. What the marketplace is

A catalog of **Voices** (not models, not artifacts) published by **Creators**, that any user
can browse, preview, and **use** in generation. Using a marketplace voice generates
**royalties** for its creator. The marketplace is the network-effect engine of the ecosystem
layer.

Critically: a listed Voice is **model-agnostic**. A buyer using `voice_X` with
`omnivoice-base` and another using it with `kokoro` both consume the same identity; the
platform resolves each one's `VoiceVariant` transparently. The marketplace never exposes
variants.

## 2. Listing model

A `MarketplaceListing` wraps a `Voice` (it does not duplicate it):

```
Voice (identity, owned by Creator)
  └── MarketplaceListing
        ├── status: draft | published | unlisted | removed
        ├── category + tags
        ├── pricing { model: free | per_use | subscription, rate }
        ├── preview_audio  (canonical, model-agnostic)
        └── stats { plays, uses, favorites, revenue }
```

Unlisting hides a voice from discovery but **preserves it and its royalty history** —
listings are soft-removed, never hard-deleted, so accounting stays intact (see
[Data §7](03-DATA_ARCHITECTURE.md)).

## 3. Publish flow

```
Creator (verified)                         [verification gate → Creator system]
  └─ select Voice → set category/tags/pricing/preview
        ▼
  validation: ownership/consent confirmed, preview present, ≥1 ready VoiceVariant
        ▼
  status = published → indexed for discovery → voice.published event
```

The `is_public` / `is_community_voice` flags that already exist on the voice become the
gate the publish flow flips. Preview audio is generated by the onboarding pipeline.

## 4. Discovery

- **Search & filter run on derived `characteristics`** (gender, age, accent, pitch, style,
  language) + text search over name/description/tags — the same structured approach the Voice
  Library already uses, scaled with conventional indexing (Postgres FTS or an external search
  index for text). **No vector search / pgvector** (see [Data §6](03-DATA_ARCHITECTURE.md)).
- Facets: language, gender, age, accent, style, model-compatibility, price, popularity.
- Sort: trending (recent uses), top (all-time uses/revenue), newest, favorites.
- Ranking inputs come from `stats` + `usage_count`; no ML ranking in early phases.

## 5. Preview & use

- **Preview:** stream the canonical `preview_audio` — no generation, no cost, no model
  selection. Model-agnostic by design.
- **Use:** the buyer calls `POST /v1/speech/generate { voice: voice_X, model: … }`. The
  platform:
  1. authorizes use (public listing → allowed; pricing/quota checked),
  2. resolves/builds the `VoiceVariant` for the chosen model,
  3. runs inference,
  4. on success, **accrues a royalty** to the listing's creator via the
     `generation.completed` → `royalty.accrued` path.

Using a marketplace voice **never requires the buyer to obtain artifacts** — they reference
the `public_voice_id` only.

## 6. Royalty-on-use loop

```
generate (marketplace voice) ──► generation.completed
        ▼
royalty engine: split gross → { infra, platform, creator }     [Monetization]
        ▼
write Royalty row (accrued) + Transaction (royalty_accrual)
        ▼
periodic settlement → Payout (Stripe Connect)  → creator console
```

Royalty **rate/terms** come from the voice's `royalty_config` (schema-ready field) and/or the
listing `pricing`. Split mechanics live in [Monetization](07-MONETIZATION_ARCHITECTURE.md).

## 7. Trust, safety & moderation

- **Ownership/consent** is enforced at onboarding and re-affirmed at publish (creator
  verification + the existing [VOICE_USAGE_POLICY](../../VOICE_USAGE_POLICY.md)).
- **Moderation:** admin review queue for new public listings; takedown = unlist + freeze
  accruals. Reported voices enter the same queue.
- **Provenance:** every listing ties to a verified `Creator`; the `public_voice_id` gives a
  stable handle for disputes/takedowns across the platform and SDKs.

## 8. CE posture

In CE the marketplace tables exist but no marketplace router is mounted and `feature.marketplace`
is off. A **future, optional** CE capability (roadmap, not committed) is **read-only browse +
import** of public marketplace voices for local use — earning no royalties locally, since the
generation happens on the user's own hardware. Creator payouts/credits/billing remain strictly
Cloud. This was the chosen open-core boundary (CE = infrastructure, Cloud = ecosystem).
